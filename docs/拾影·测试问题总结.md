## 高频问题类型分析

### 1. Qt 布局系统问题（6个）

**核心教训**：QGridLayout 的 `removeWidget()`/`takeAt()` 会导致内部状态残留，反复添加移除后布局计算不可靠。

**典型案例链**：
- #16 用 QGridLayout 的 `addWidget(card, row, col)` 定位 → 少图片时不跟随
- #22 改用 `takeAt(0)` 清理 → 第一行仍无法点击
- #23 放弃 QGridLayout，改用手动 `setGeometry()` → 解决

**面试要点**：
- Qt 的 `QGridLayout` 有内部状态（行/列 stretch、minimum size 等），`removeWidget` 不会完全重置
- `setWidgetResizable(True)` 会让滚动区域拉伸子组件，需要 `SizePolicy.Maximum` + `setFixedHeight` 配合
- `scroll_area.width()` 包含滚动条宽度，`scroll_area.viewport().width()` 才是实际内容区宽度

### 2. 跨线程信号与 UI 更新（4个）

**核心教训**：`QApplication.processEvents()` 在 macOS 上会导致 `CATransaction synchronize` 崩溃，应避免在信号处理函数中调用。

**典型案例链**：
- #21 加载完不显示 → 添加 `processEvents()` 强制刷新
- #25 加载时不实时显示 → 添加 `processEvents()` → macOS 崩溃
- 最终方案：依赖 Qt 事件循环自然处理 paint 事件，不用 `processEvents()`

**面试要点**：
- PyQt 的跨线程信号使用 `Qt.QueuedConnection`，信号作为事件插入主线程事件队列
- `processEvents()` 会递归处理事件，可能导致重入（re-entrancy）
- 正确做法：让事件循环自然驱动 UI 更新，而非强制插入事件处理

### 3. 异常处理缺失导致崩溃（5个）

**核心教训**：外部 API 调用必须有完整的异常处理链（try-catch-finally），包括网络异常、JSON 解析异常、资源释放。

**典型案例**：
- #2 测试连接崩溃 → 缺少 httpx 异常捕获
- #8 空地址崩溃 → 只检查空字符串，未校验 URL 格式
- #9 DeepSeek 崩溃 → 模型名硬编码 + JSON 解析无容错

**面试要点**：
- 防御式编程：在系统边界（用户输入、外部 API）必须校验
- 资源释放用 `finally` 或 `async with`
- 熔断机制：连续失败 N 次自动停止，避免无效重试

### 4. 图像格式兼容性（2个）

**核心教训**：PIL 的 RGBA 模式不能直接保存为 JPEG，需要先转 RGB。

**面试要点**：
- JPEG 不支持 alpha 通道，PNG 支持
- `Image.convert("RGB")` 会丢弃 alpha 通道
- 处理用户上传的图片时，必须考虑所有颜色模式（RGB/RGBA/P/LA/CMYK）

### 5. 状态管理混乱（2个）

**核心教训**：切换模式时不应覆盖已有的状态数据，应将来源和数值分开存储。

**典型案例**：#7 评分来源被覆盖 → 新增 `_score_sources` 字典独立记录来源

**面试要点**：
- 状态数据按来源维度分离存储，不要用单一变量动态推断
- 切换上下文时，已有数据不应被修改

## 修复策略总结

| 策略 | 适用场景 | 本项目案例 |
|------|----------|------------|
| 放弃框架，手动控制 | 框架内部状态不可控 | QGridLayout → setGeometry |
| 分离存储 | 状态来源多样 | _scores + _score_sources |
| 防御式校验 | 用户输入/外部 API | URL 格式、模型名、连续失败熔断 |
| 事件循环信任 | UI 实时更新 | 移除 processEvents，依赖自然刷新 |
| 滚动容器 | 内容超出可视区域 | 侧边栏 QScrollArea |
| 类型转换兼容 | 跨格式处理 | RGBA→RGB、JSON 容错解析 |
